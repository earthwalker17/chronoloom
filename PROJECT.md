# ChronoLoom — Full Project Context

## 1. Project Summary

ChronoLoom is an AI-native alternate-life simulator.

The player enters a historical or future era, receives or chooses a concrete identity, makes meaningful choices inside a visually presented world, and watches an AI-driven simulation infer the next stage of that life.

The project evolved from an earlier concept called “Time Travel Test,” which was originally a lightweight personality-style experience:

- User selects a historical or future era.
- User selects a broad role or goal.
- AI generates a “letter of understanding” and personality interpretation.

ChronoLoom is a major upgrade from that original idea.

It should not be a one-time quiz. It should become a playable, stateful, visual, AI-driven life simulation system.

The emotional goal is simple:

> Let the user feel they have briefly lived another life.

---

## 2. Why This Project Matters

The project is based on a strong human desire:

> What if I had been born in another time, another place, another body, another social role?

Many users do not only want to know their “personality type.” They want to experience a different possible life.

Examples:

- If I lived in Tang Chang’an, what kind of person would I become?
- If I were a Renaissance apprentice, would I pursue art, money, faith, or survival?
- If I were born in a Mars colony, would I become a builder, rebel, scientist, bureaucrat, explorer, or exile?
- If I were not myself, what would remain recognizably mine?

This makes ChronoLoom more than entertainment. It can combine:

- Curiosity
- Roleplay
- Historical imagination
- Future speculation
- Self-exploration
- Narrative identity
- Consequence-driven simulation

The most valuable product direction is not “make the original test bigger.”

The more powerful direction is:

> Turn a static personality test into an enterable, selectable, consequence-driven alternate-life simulation.

---

## 3. Core Product Definition

ChronoLoom is an:

- Interactive alternate-life simulator
- AI-powered historical/future world system
- Dynamic life trajectory generator
- Visual narrative simulation product
- Lightweight game-like experience

It should borrow from games, but it is not a conventional game.

It should borrow from personality tests, but it is not a test.

It should borrow from interactive fiction, but it should not be a fixed branching novel.

It should borrow from chatbots, but it should not feel like a plain chatbot.

Its defining feature is:

> The next stage of the user’s life should be inferred from structured world state, identity state, NPC state, relationship state, historical constraints, and player actions.

---

## 4. Core Difference From Ordinary Story Games

Ordinary story games usually rely on pre-authored branches.

ChronoLoom should rely on a stateful simulation loop.

The system should not simply ask:

> Which branch did the user choose?

It should ask:

- What did the user do?
- What does that reveal about their values, ambition, fear, loyalty, curiosity, risk tolerance, or moral boundaries?
- What changed in the world?
- Which NPCs noticed?
- Which resources increased or decreased?
- Which risks were created?
- Which future opportunities opened?
- Which paths became less likely?
- What is the most plausible next scene?

The player should feel that the world remembers them.

---

## 5. Core Difference From Ordinary Chatbots

ChronoLoom should not feel like chatting with an AI narrator.

The product should feel like entering a world.

The AI is not the product by itself. The AI is the engine behind:

- Scene generation
- Consequence reasoning
- NPC reaction
- Timeline formation
- World evolution
- Final life reflection

The user-facing experience should include visual structure, state, timeline, identity, and world feedback.

A plain text box alone is not enough for the desired product identity.

The interaction surface is the scene itself: the people of the world are visible, approachable and clickable, and talking to them is a bounded, structured act inside the simulation — never an open chat box.

---

## 6. Core Difference From the Original Time Travel Test

The original concept solved a light self-understanding need:

> I want to know which era or identity resonates with me.

ChronoLoom solves a deeper experiential need:

> I want to enter another world, live as someone else, make choices, and see what kind of life emerges.

This changes the product from:

- Result generation
- Static interpretation
- One-time emotional output

into:

- State progression
- Dynamic world response
- Long-form identity simulation
- Replayable alternate lives
- Personal life trajectory archive

The final reflection still matters, but it should come after lived choices, not replace them.

---

## 7. The Mother Experience Loop

The product must protect this loop:

1. Enter an era.
2. Choose or receive a concrete identity.
3. Arrive in a scene.
4. Take action.
5. Simulate consequences.
6. Generate the next stage.
7. Record the life trajectory.

Each step should be concrete.

### 7.1 Enter an Era

The user is introduced to a specific historical or future world.

The era should have:

- Atmosphere
- Institutions
- Conflicts
- Opportunities
- Social constraints
- Visual identity
- Plausibility boundaries

### 7.2 Choose or Receive an Identity

The user should not be an abstract observer.

They should become someone concrete:

- A scholar
- A merchant apprentice
- A palace attendant
- A migrant worker
- A monk
- A mapmaker
- A sailor
- An engineer
- A court clerk
- A Mars colony maintenance worker
- A biotech apprentice
- A climate refugee organizer

Identity should determine:

- Access
- Social position
- Resources
- Obligations
- Risks
- Relationships
- Possible futures

### 7.3 Arrive in a Scene

A scene should feel spatial and atmospheric.

Even if full 3D is not available yet, the user should feel they are somewhere.

Scene presentation may use:

- Real-time 3D
- Pseudo-3D diorama
- Stylized maps
- Generated scene cards
- Character portraits
- Location panels
- Lighting/camera effects
- Environmental text

### 7.4 Take Action

Choices should express something about the user.

Good choices may reveal:

- Ambition
- Caution
- Loyalty
- Curiosity
- Compassion
- Ruthlessness
- Survival instinct
- Love of knowledge
- Desire for recognition
- Risk appetite
- Relationship priorities

Avoid choices that are merely cosmetic.

### 7.5 Simulate Consequences

After each action, update state.

Possible consequence types:

- Resource changes
- Reputation changes
- Trust changes
- Health changes
- Skill changes
- Relationship changes
- Institutional attention
- Opportunity unlocked
- Risk increased
- Time advanced
- New rumor added
- Future event triggered

### 7.6 Generate the Next Stage

The next scene should emerge from state.

It should not feel random.

The system should produce:

- A coherent next situation
- Meaningful tension
- Contextual options
- Updated world hints
- Visible consequences from previous actions

### 7.7 Record the Life Trajectory

The system should keep a timeline.

A good timeline includes:

- Major decisions
- Consequences
- Relationship shifts
- Location changes
- Turning points
- Missed opportunities
- Achievements
- Failures
- Final life arc

The user should be able to look back and feel: “This was my version of that life.”

## 9. Short-Term Prototype Target

The current near-term goal is a playable public vertical slice.

A good first version may contain:

- One era
- One compact district or location cluster
- Three to five identities
- Several scenes
- A few recurring NPCs
- A simple but structured world state
- A simple player state
- AI-generated next scenes
- Consequence tracking
- Timeline tracking
- A final life report
- A strong visual front-end
- A basic 3D or pseudo-3D presentation layer
- Local run instructions
- Simple deployment path

The goal is not to build a complete open world.

The goal is to prove that the central loop has magic.

## 10. Recommended First-Era Strategy

Do not build many eras at once.

Pick one era with:

- Strong visual identity
- Strong social tension
- Clear identity roles
- Concrete constraints
- Good demo value
- Easy-to-understand stakes

Possible first-era candidates:

### Tang Chang’an

Strengths:

- Visually rich
- Cosmopolitan
- Strong class/status dynamics
- Poets, merchants, monks, officials, performers, foreigners, artisans
- Good for Chinese cultural imagination and social sharing

Possible compact setting:

> A market district near the eastern gate of Chang’an during several decisive days.

### Renaissance Florence

Strengths:

- Artists, bankers, guilds, patrons, religious tension
- Strong craft/apprentice identities
- Good Western cultural familiarity
- Visual richness

Possible compact setting:

> A workshop district between guild politics, patronage, and church influence.

### Northern Song Bianjing

Strengths:

- Urban prosperity
- Civil examination pressure
- Commerce and entertainment
- Scholar-official culture
- Street life and class mobility

Possible compact setting:

> A neighborhood near a bridge market during festival season.

### Age of Exploration Port City

Strengths:

- Maps, ships, trade, risk, imperial ambition
- Cultural encounter
- Moral ambiguity

Possible compact setting:

> A port tavern, dock, map room, and merchant house around one voyage decision.

### Industrial Revolution City

Strengths:

- Invention, factory labor, class conflict, journalism, capital
- Clear modernity tension

Possible compact setting:

> A factory district during a strike, invention race, or public scandal.

### Near-Future Mars Colony

Strengths:

- Strong speculative appeal
- Survival constraints
- Governance, ecology, AI, engineering, rebellion
- Easier to invent within plausible bounds

Possible compact setting:

> A dome district during an oxygen, governance, or infrastructure crisis.

The first era should be selected based on prototype feasibility and emotional distinctiveness.

## 11. Core System Concepts

ChronoLoom should be state-driven.

These concepts do not need complex implementations at first, but they should shape the code.

### 11.1 Era Bible

A structured source of truth for the selected era.

It may include:

- Era name
- Time period
- Location
- Political structure
- Economy
- Social classes
- Gender/class constraints
- Major institutions
- Common roles
- Technology level
- Taboo behaviors
- Religious/philosophical assumptions
- Visual motifs
- Tone
- Plausibility rules
- Common dangers
- Common opportunities

Purpose:

- Prevent generic AI drift
- Keep scenes grounded
- Give identity and choice meaning

### 11.2 Player State

Represents the player’s alternate-life identity.

Possible fields:

- Name
- Age
- Role
- Origin
- Family background
- Social status
- Skills
- Resources
- Health
- Reputation
- Ambition
- Values
- Hidden tendencies inferred from choices
- Relationships
- Current location
- Current life stage
- Major decisions
- Open opportunities
- Current risks

### 11.3 World State

Represents the current world situation.

Possible fields:

- Current date or chapter
- Location
- Public mood
- Faction tensions
- Economic conditions
- Rumors
- Active events
- Institutional pressure
- Weather or environmental conditions
- Opportunities available
- Threats emerging

### 11.4 NPC State

Important NPCs should have persistent state.

Possible fields:

- Name
- Role
- Social position
- Motivation
- Relationship to player
- Trust level
- Fear/respect/resentment
- Memory of interactions
- Current goal
- Possible future action

A few strong NPCs are better than many shallow ones.

### 11.5 Event Queue

The system should maintain possible upcoming events.

Events may be triggered by:

- Player decisions
- Time passing
- NPC actions
- World state
- Historical constraints
- Randomness within bounded plausibility
- Narrative pacing needs

Events should be explainable from current state.

### 11.6 Causal Ledger

Record why changes happened.

Example entries:

- Player defended NPC A in public.
- NPC A trust increased.
- Local official suspicion increased.
- Future opportunity opened: private meeting with NPC A.
- Future risk opened: official surveillance.

This improves coherence and final reports.

### 11.7 Narrative Director

The AI layer that presents scenes.

It should:

- Read structured state
- Generate concise immersive prose
- Present meaningful context
- Offer strong choices
- Preserve tone
- Avoid over-explaining mechanics
- Avoid ignoring state

### 11.8 Simulation Director

The stricter logic layer that updates state.

It should:

- Parse player action
- Apply rules and plausibility
- Update player/world/NPC/event state
- Advance time if appropriate
- Record causal changes
- Pass updated state to the Narrative Director

### 11.9 Visual Director

The layer that makes the world visible.

It may control:

- 3D scene parameters
- Camera
- Lighting
- Weather/mood
- Props
- Location cards
- Character portraits
- Map markers
- Timeline visuals
- Final result cards

The first version can be simplified, but the product should not look like a plain chatbot.

## 12. AI Generation Principles

The AI should generate from state, not from nothing.

Preferred flow:

1. Read current structured state.
2. Read player action.
3. Apply plausibility and safety boundaries.
4. Update structured state.
5. Produce a concise immersive scene.
6. Offer next actions.
7. Save changes.

The simulation step and narrative step can be separated.

For example:

- Simulation update returns structured JSON.
- Narrative generation renders that JSON into user-facing scene text.
- Visual layer uses state to render/update scene.

Avoid:

- Infinite unconstrained freeform generation
- Random dramatic twists
- Generic fantasy content
- Inconsistent history
- Long exposition dumps
- Modern phrasing in historical worlds unless intentional
- Choices that all converge immediately
- Hidden prompt-only state that cannot be inspected or persisted

## 13. Product Taste

The tone should be:

- Immersive
- Specific
- Human
- Historically or futuristically textured
- Slightly poetic but not purple
- Serious enough to feel meaningful
- Playful enough to invite exploration
- Reflective but not preachy

Avoid:

- Generic AI fantasy language
- “You feel a sense of destiny” clichés
- Overly moralized endings
- Shallow personality-test analysis
- Random prophecies
- Excessively long scenes
- Choices with obvious “correct” answers
- Exposition that reads like a textbook

---

## 14. Visual Experience Direction

The scene view should feel central.

Even if the first version is lightweight, the user should feel spatial presence.

Possible visual forms:

- 3D diorama
- Small explorable scene
- Generated environment cards
- Stylized map
- Character portrait panels
- Location-specific mood lighting
- Scene transition effects
- Timeline/path graph
- Fate card
- Alternate-life archive card

A good first visual target:

> A compact, atmospheric scene that changes enough to reflect life progression — and that the player acts THROUGH: characters are figures you can walk up to, click, and talk with, not portraits beside a text panel.

Avoid spending all effort on visuals while neglecting the state loop.

The product needs both:

- Visual immersion
- Consequence-driven simulation

## 16. Final Life Report

The final report is important because it connects the alternate life back to the user.

It may include:

- Life title
- Identity summary
- Major turning points
- Relationships that shaped the life
- Values revealed through choices
- Risks taken or avoided
- What the user protected
- What the user sacrificed
- What kind of person they became in that world
- A poetic closing letter or archive note
- Shareable result card

The final report should be grounded in actual timeline data.

It should not be generic personality-test filler.

ChronoLoom is naturally shareable if the output is specific.

Examples of shareable hooks:

- “I lived seven years as a Tang dynasty market scribe and accidentally became a forbidden poet.”
- “In Renaissance Florence, I started as a workshop apprentice and ended as a patron’s spy.”
- “My Mars colony life ended because I chose public trust over oxygen ration secrecy.”
- “This AI world made me live another life and then showed me what my choices revealed.”

For the prototype, even a simple generated card or screenshot-ready final report can help.

## 20. Implementation Direction

No fixed technology stack is required.

Possible practical direction for a rapid prototype:

- Browser-first app
- Lightweight 3D using a web-friendly renderer if feasible
- Structured state in JSON, SQLite, or similar
- AI provider abstraction
- Local-first development
- Clear separation between simulation and presentation
- Simple persistence for user sessions
- Exportable timeline/report data

Architecture should avoid mixing everything into one prompt.

Recommended separation:

- UI layer
- Scene/visual layer
- Player/world/NPC state layer
- Simulation update layer
- Narrative generation layer
- Persistence layer
- Timeline/report layer
- AI provider layer

## 23. Scope Control

The project is extremely easy to over-expand.

Watch for these traps:

- “Let’s add ten eras before the first one works.”
- “Let’s make a complete open world before choices matter.”
- “Let’s build a beautiful 3D city with no simulation.”
- “Let’s write huge lore files with no playable loop.”
- “Let’s add NPC autonomy before we have persistent NPC state.”
- “Let’s implement every RPG system.”
- “Let’s make AI generate everything with no validation.”
- “Let’s build a personality test again and forget the life simulation.”

The correct first milestone:

> One user, one identity, one world, several choices, visible consequences, timeline, final life report.

## 24. Minimum Demo-Ready Criteria

The prototype is demo-ready only when:

- A user can start a new session.
- The user can choose or receive a specific identity.
- The user can enter a visually presented scene.
- The user can take meaningful actions.
- Structured state updates after actions.
- The next scene reflects previous actions.
- At least one recurring NPC or relationship changes.
- A timeline records major events.
- The final life report reflects actual timeline data.
- The app can be run locally with clear instructions.
- Main flow works without obvious breakage.
- Visual presentation has enough atmosphere for screenshots.
- The system avoids obvious AI drift.
- The system is not just a chatbot.

## 26. Product North Star

ChronoLoom is not about simulating all of history.

It is about making one alternate life feel specific, consequential, and personal.

The product should make the user wonder:

> If I had lived there, would I still have chosen the same kind of life?

That question is the heart of the project.
